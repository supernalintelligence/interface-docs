## The Testing Tax

Every CLI command you write needs tests. Required parameter validation. Default value checks. Enum constraints. Type validation. Range boundaries. It's tedious, repetitive, and easy to forget.

```typescript
// Your command
const listUsersCommand = new UniversalCommand({
  name: 'list users',
  input: {
    parameters: [
      { name: 'status', type: 'string', enum: ['active', 'inactive'] },
      { name: 'limit', type: 'number', default: 50, min: 1, max: 1000 },
      { name: 'format', type: 'string', required: true }
    ]
  },
  handler: async (args) => { /* ... */ }
});

// Tests you have to write manually 😩
describe('list users', () => {
  it('should require --format', () => { /* ... */ });
  it('should default limit to 50', () => { /* ... */ });
  it('should reject invalid status enum', () => { /* ... */ });
  it('should reject limit below 1', () => { /* ... */ });
  it('should reject limit above 1000', () => { /* ... */ });
  // ... 15 more tests
});
```

But wait—the schema already **describes** all these constraints. Why are we writing tests for things we've already declared?

## The Realization

Look at that schema again:

```typescript
{ name: 'limit', type: 'number', default: 50, min: 1, max: 1000 }
```

This single line tells us:
- ✅ `limit` should default to `50`
- ✅ Values below `1` should fail
- ✅ Values above `1000` should fail
- ✅ Non-numbers should fail (or be coerced)

**The schema IS the test specification.** We just need to read it.

## Introducing JestTestGenerator

We built a generator plugin that introspects UniversalCommand schemas and emits comprehensive Jest test suites:

```typescript
import { JestTestGenerator } from '@supernal/universal-command/generators';

const generator = new JestTestGenerator();
const tests = generator.generateContent(listUsersCommand, {
  outputDir: './tests/generated'
});
```

Output:

```typescript
/**
 * Auto-generated tests for: list users
 * Generated by: @supernal/universal-command (jest-tests plugin)
 */

describe('list users', () => {
  describe('required parameters', () => {
    it('should require --format', async () => {
      const result = listUsersCommand.validateArgs({});
      expect(result.valid).toBe(false);
      expect(result.errors).toContainEqual(
        expect.objectContaining({
          path: 'format',
          message: expect.stringContaining('required')
        })
      );
    });
  });

  describe('default values', () => {
    it('should default limit to 50', () => {
      const result = listUsersCommand.validateArgs({ format: 'json' });
      expect(result.valid).toBe(true);
      expect(result.data?.limit).toBe(50);
    });
  });

  describe('enum validation', () => {
    it('should accept valid enum value for status', () => {
      const result = listUsersCommand.validateArgs({
        format: 'json',
        status: 'active'
      });
      expect(result.valid).toBe(true);
    });

    it('should reject invalid enum value for status', () => {
      const result = listUsersCommand.validateArgs({
        format: 'json',
        status: '__invalid__'
      });
      expect(result.valid).toBe(false);
      expect(result.errors).toContainEqual(
        expect.objectContaining({
          message: expect.stringContaining('must be one of')
        })
      );
    });
  });

  describe('number range validation', () => {
    it('should reject limit below minimum (1)', () => {
      const result = listUsersCommand.validateArgs({
        format: 'json',
        limit: 0
      });
      expect(result.valid).toBe(false);
    });

    it('should reject limit above maximum (1000)', () => {
      const result = listUsersCommand.validateArgs({
        format: 'json',
        limit: 1001
      });
      expect(result.valid).toBe(false);
    });
  });

  // ... CLI tests, execution tests, etc.
});
```

**From one schema definition, we generate 15-20 tests automatically.**

## The Meta Part: A Command That Tests Itself 🐕

Here's where it gets fun. We needed a CLI command to run this generator:

```bash
sc generate tests --output ./tests/generated
```

So we wrote it... as a UniversalCommand:

```typescript
export const generateTestsCommand = new UniversalCommand({
  name: 'generate tests',
  description: 'Auto-generate Jest tests from UniversalCommand schemas',
  input: {
    parameters: [
      { name: 'output', type: 'string', default: './tests/generated' },
      { name: 'commandsImport', type: 'string', default: '../commands' },
      { name: 'cli', type: 'boolean', default: true },
      { name: 'api', type: 'boolean', default: false },
      { name: 'mcp', type: 'boolean', default: false },
      { name: 'dryRun', type: 'boolean', default: false },
      { name: 'typescript', type: 'boolean', default: true },
    ]
  },
  handler: async (args, context) => {
    const generator = new JestTestGenerator();
    return generator.generate(registry, args);
  }
});
```

This command:
1. **Is a UniversalCommand** with a typed schema
2. **Uses the JestTestGenerator** to generate tests
3. **Can generate tests for itself**

```
UniversalCommand → JestTestGenerator → Tests for UniversalCommands
       ↑___________________________________|
                    (dogfooding)
```

And yes, we tested it:

```typescript
describe('generate tests command', () => {
  it('should execute with dry run', async () => {
    const result = await generateTestsCommand.execute(
      { output: './tests', dryRun: true },
      { interface: 'test' }
    );
    expect(result.filesGenerated).toBeGreaterThanOrEqual(0);
  });
});
```

**38 tests pass. The command that generates tests, tested by tests it could generate.**

## What Gets Generated

For each command, the generator creates tests for:

| Category | What's Tested |
|----------|---------------|
| **Schema** | Name, description, handler exist |
| **Required params** | Missing required args fail validation |
| **Defaults** | Default values are applied correctly |
| **Enums** | Valid values pass, invalid values fail |
| **Number ranges** | Min/max constraints enforced |
| **Type validation** | Type checking works |
| **CLI interface** | Command executes via CLI |
| **API interface** | (optional) REST endpoint works |
| **MCP interface** | (optional) Tool conversion works |
| **Execution** | Happy path completes |

## Usage

```bash
# Generate tests for all registered commands
sc generate tests

# Custom output directory
sc generate tests --output ./tests/commands

# Include API and MCP interface tests
sc generate tests --api --mcp

# Preview without writing files
sc generate tests --dry-run

# JavaScript instead of TypeScript
sc generate tests --typescript false
```

Or use the generator directly:

```typescript
import { JestTestGenerator, CommandRegistry } from '@supernal/universal-command';

const generator = new JestTestGenerator();
const registry = new CommandRegistry();

// Register your commands
registry.register(myCommand);
registry.register(anotherCommand);

// Generate all tests
const result = await generator.generate(registry, {
  outputDir: './tests/generated',
  includeCLI: true,
  includeAPI: true
});

console.log(`Generated ${result.files.length} test files`);
```

## The Philosophy

This is part of UniversalCommand's broader vision: **define once, generate everything**.

| You Define | We Generate |
|------------|-------------|
| Command schema | CLI interface |
| | REST API endpoints |
| | MCP tool definitions |
| | OpenAPI spec |
| | **Jest tests** ← NEW |

Your schema is the source of truth. Everything else flows from it.
